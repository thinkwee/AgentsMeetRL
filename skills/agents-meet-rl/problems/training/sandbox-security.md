# Question: how to isolate untrusted code execution during agentic RL

## When this matters

If your env executes code generated by the policy:

- Code/SWE agents (SWE-Gym, SWE-Swiss, swe-rl).
- Tool-integrated reasoning with Python interpreter (ReTool).
- Math agents that run user-supplied code.
- CUDA agents (CUDA-L1, CUDA-Agent).
- Browser agents with arbitrary JS injection capability.

The policy is **adversarial by definition** during exploration. Even
with a benign final task, intermediate rollouts can:

- Probe the host filesystem.
- Open network connections.
- Exhaust resources (forkbomb, infinite memory).
- Read other rollouts' state (cross-contamination).
- Read training data / model weights.

These are not theoretical: agentic RL has produced rollouts that
inadvertently `rm -rf /tmp` because some intermediate state matched a
"cleanup" pattern.

## Threat model

Different from production "agent runs user code" where the user is
adversarial. Here:

- The policy is your model — not directly malicious but exploring.
- The "user" is the RL training loop.
- Data corruption / privilege escalation is possible but rare.
- Resource exhaustion (DoS-style) is **common**.

Defense should focus on:

- Bounding resource use (CPU, memory, time).
- Filesystem isolation.
- Network egress control.
- Per-rollout isolation (no cross-contamination).

## Levels of isolation

### Level 0: nothing (don't do this)

Run code in the trainer process. One bad rollout takes the trainer
down. Used by hobby projects only.

### Level 1: subprocess + rlimit

```python
import resource
def run_code(src, timeout=10):
    def setlimits():
        resource.setrlimit(resource.RLIMIT_CPU, (10, 10))
        resource.setrlimit(resource.RLIMIT_AS, (1<<30, 1<<30))  # 1GB
    return subprocess.run(
        [sys.executable, "-c", src],
        timeout=timeout,
        preexec_fn=setlimits,
        capture_output=True,
    )
```

Provides resource caps but no filesystem / network isolation. Bypassed
trivially by `os.system`. Only OK for fully-trusted code.

### Level 2: chroot / unshared namespaces

`unshare -n` (no network), `unshare -m` (private mount), bind-mount
read-only root. Linux primitives.

Better. Still shares kernel; container escape is possible (rare).

### Level 3: Docker container

```python
docker.run(
    image="python:3.11-slim",
    command=["python", "-c", src],
    network_mode="none",
    read_only=True,
    cap_drop=["ALL"],
    mem_limit="1g",
    cpus="1.0",
    pids_limit=64,
    timeout=30,
    remove=True,           # destroy container after
)
```

Container per rollout. Most agentic-RL projects in the corpus use this.
Trade-off: container spin-up is slow (1-3 seconds). Pool / pre-warm
containers to amortize.

### Level 4: gVisor or Firecracker (microVM)

User-space kernel (gVisor) or microVM (Firecracker). Stronger than
Docker; fewer kernel exposures.

```yaml
# Firecracker example: each rollout gets a tiny VM
runtime: firecracker
vm_kernel: minimal-kernel-5.10.0
rootfs: minimal-rootfs.ext4
mem_size_mib: 512
```

Trade-off: more setup; higher memory overhead per VM.

Used by some production deployments (less common in research).

### Level 5: separate machine

Air-gapped execution server, no shared filesystem. Heavyweight; only
when threat model demands it.

## Recommended setup for agentic RL

For most code-agent training:

- **Level 3 (Docker)** with strict caps, `network_mode=none` unless
  network is required by the task, read-only filesystem, ephemeral
  per-rollout container.
- **Pool of pre-warmed containers** to mitigate spin-up cost.
- **Resource limits**: 1 CPU, 1GB RAM, 30s timeout, 64 PIDs.

```python
class SandboxPool:
    def __init__(self, size=64):
        self.containers = [self._make_container() for _ in range(size)]

    def execute(self, code, timeout=30):
        c = self.containers.pop()
        try:
            result = c.exec_run(["python", "-c", code], timeout=timeout)
            return result
        finally:
            c.restart()  # reset state
            self.containers.append(c)
```

## Filesystem isolation

For SWE agents that need a working filesystem:

- Per-rollout tmpdir bind-mounted.
- All paths outside the tmpdir denied.

```python
docker.run(
    image="...",
    volumes={
        f"/tmp/rollout_{rollout_id}": {"bind": "/workspace", "mode": "rw"}
    },
    working_dir="/workspace",
    read_only=True,                 # base FS read-only
    tmpfs={"/tmp": "rw,size=128M"}, # tmpfs for /tmp only
)
```

After rollout: `rm -rf /tmp/rollout_<id>`.

> paper: `https://arxiv.org/abs/2412.21139`, UC Berkeley/UIUC/CMU/Apple,
> date: 2024.12. Why: per-task Docker isolation with proper
> filesystem bounds; reference implementation.

> **R2E-Gym** — github: `https://github.com/R2E-Gym/R2E-Gym`,
> UC Berkeley/ANU, date: 2025.4. Why: alternative SWE sandbox with
> different isolation primitives.

## Network policy

Default: **deny everything**.

Exceptions only for tools that genuinely need network (search agents
calling APIs):

- Allowlist by hostname (DNS-based).
- Rate-limit per-rollout.
- Log every external call.
- Use a forward proxy that enforces the allowlist:

  ```python
  docker.run(
      ...,
      environment={
          "HTTPS_PROXY": "http://proxy.internal:8080",
      },
      # Container can't reach internet directly, only through proxy.
  )
  ```

Prevents accidental data exfiltration if the policy outputs sensitive
strings; prevents DoS on third-party services.

## Resource exhaustion patterns to bound

- **CPU**: `cpus="1.0"` (1 core), `cpu_period`/`cpu_quota`.
- **Memory**: `mem_limit="1g"`, `memswap_limit="1g"` (no swap).
- **PIDs**: `pids_limit=64` (forkbomb defense).
- **File descriptors**: `ulimit -n 256` inside container.
- **Wall-clock**: `timeout=30` on the docker.run.
- **Output size**: cap stdout at 1MB.

```python
proc = subprocess.run(...)
stdout = proc.stdout[:1024*1024]  # 1MB cap
```

## What sandbox doesn't fix

### The model can still produce malicious *content*

Sandbox prevents the *code* from doing damage. The *output content*
(text the model writes) can still be:

- Personally identifying / private (if it learned from training data).
- Harmful instructions.
- Prompt injection attacks against downstream consumers.

That's a content-safety problem, not a sandbox problem. See
`reward-up-eval-flat.md` and the `safety` references.

### Side-channel timing attacks

A sandboxed container can still observe:

- Wall-clock time (for timing-side-channel inference).
- Cache timing.

Not a concern for typical agentic RL but worth flagging for
security-critical deployments.

### Container escape (rare but real)

Docker has had escape CVEs. gVisor / Firecracker are stronger but
still imperfect. For high-stakes deployments, use a separate machine.

## Sandbox flakiness vs sandbox security

Sandbox flakiness (`env-flakiness.md`) is about:

- Container OOM under normal use.
- Port conflicts.
- File descriptor leaks.
- Network hiccups.

Sandbox security is about:

- Adversarial code containment.
- Cross-rollout isolation.
- Resource exhaustion attacks.

Both are real concerns. Address both.

## Per-task cleanup

Always:

```python
def cleanup_rollout(rollout_id):
    container.kill()
    container.remove()
    shutil.rmtree(f"/tmp/rollout_{rollout_id}")
```

Failure to cleanup → disk full, container limits hit, training stalls
mid-run.
